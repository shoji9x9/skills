## 事象

PR #31 の Copilot レビューで、`skills/kaizen/scripts/kaizen-archive.sh` が `.kaizen/archive/` 配下の
ファイルを引数に渡されると `mv` / `git mv` が**同一ディレクトリへの移動**で失敗し、`set -e` により
**最後の `INDEX.md` 再生成（クリーンアップ相当）に到達せず中断**する、と指摘された。実際にその通りで、
既アーカイブ済みパスを渡すと索引が更新されないまま異常終了していた。

## 根本原因

`set -e` のスクリプトで「副作用の最後にクリーンアップ／索引再生成を置く」設計だが、途中に
**失敗・no-op になりうる操作**（既に目的状態にある入力＝同一ディレクトリへの mv、既存先への移動など）が
あると、`set -e` が cleanup へ到達する前に抜けてしまう。入力の**冪等性**（既処理対象のスキップ）を
保証していなかったのが根本原因。「成功条件＝最後まで到達」を、途中の操作の成否に暗黙依存させていた。

KEDB 照合: `[[2026-06-11-extractor-session-rearms-sentinel]]` の「書けない副作用を成功条件にしない」と
同系。今回は `set -e` × 末尾クリーンアップの**到達性**という新形態。

## 提案

ファイルを移動して最後に索引・集計を作り直す系の bash スクリプトは:

1. **既に目的状態にある入力を明示スキップ**して冪等にする（今回の修正＝親ディレクトリが archive と
   一致する引数を skip）。`mv`/`cp` の no-op・同一パスで `set -e` が中断しないようにする。
2. クリーンアップ／再生成の到達性が重要なら `trap '...' EXIT` 等で**途中で失敗しても実行**する形を検討する。
3. レビュー指摘（特に「常に壊れる／失敗する」系）は適用前に**使い捨て環境で再現テスト**して裏取りする。
