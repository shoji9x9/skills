# 検証コマンドの成否をパイプライン越しの `$?` で判定しない

## 事象

依存更新の検証で `pnpm audit 2>&1 | tail -15; echo "exit=$?"` を実行し、`exit=0` を確認した。
しかしこの `0` はパイプライン末尾の `tail` の終了コードであり、`pnpm audit` 本体（脆弱性検出時は非 0 を返す）
のものではない。実際には 12 件の脆弱性が残った状態で `exit=0` と表示されており、「audit が通った」と
誤読する余地があった。今回は直後に `pnpm audit --json` を jq で解析して正しく判断したため実害はなかった。

## 根本原因

- bash の `$?`（パイプライン直後）は最後のコマンドの終了コードで、`set -o pipefail` なしでは上流コマンドの
  失敗が隠れる。
- 「出力を絞りたい（tail/head/grep）」と「終了コードを確認したい」を 1 つのパイプラインで両立しようとしたが、
  既定のシェル設定では両立しない書き方だった。
- パイプで出力を整形するときの終了コード確認の規律がプロジェクトの検証ガイドラインに明文化されていない。

## 提案

検証コマンドの成否はパイプライン越しの `$?` で判定せず、`${PIPESTATUS[0]}` / `set -o pipefail` /
出力をファイルに落として分離実行（`cmd > out; echo $?; tail out`）のいずれかで対象コマンド自身の
終了コードを取る。

- 適用先候補: `AGENTS.md` の検証節（「検証は目的を果たす最低限のツール実行で行う」の周辺）に 1 行追記する。
  例: 「パイプで出力を整形する検証では、成否判定に `${PIPESTATUS[0]}` か `set -o pipefail` を使う
  （末尾 `$?` はパイプ最後のコマンドの終了コード）」
- 横断スコープ: 対話的な検証コマンドだけでなく、スキルのコマンド例・hook スクリプトにも同じ落とし穴がある。
  apply 時に `skills/**` / `.agents/skills/**` のシェル例で `| tail` / `| head` / `| grep` 直後に
  `$?` を参照している箇所がないか grep で確認する。
