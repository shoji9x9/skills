# 自律ループ中の kaizen ゲート再装填が反復ごとに commit を中断する

## 事象

PR #65 の pr-finalize-loop（自律ループ）実行中、反復 2 のレビュー対応 commit が
kaizen コミット前ゲートに再ブロックされた。同一セッション内で既に抽出を完了していた
（反復 1 開始前に `kaizen --current` 実行・センチネル削除済み）にもかかわらず、
ターン終了ごとに Stop フックが無条件でセンチネル `.kaizen/.pending-extract` を
再装填するため、次のターンで commit するたびにゲート→抽出→ユーザー承認が挟まる。

**再発（2026-07-05）**: 自律ループではない通常の対話セッション（Issue #68 着手）でも同型が発生。
セッション中盤で抽出・センチネル削除済みだったが、以降のターン終了ごとに Stop フックが再装填し、
セッション後半の commit（ユーザー明示指示）がゲートにブロックされた。自律ループ固有ではなく
「抽出後にターンをまたいで commit する全セッション」に及ぶため priority を high に引き上げ。

## 根本原因

- なぜ再ブロックされたか → 前ターン終了時の Stop フックがセンチネルを書いた。
- なぜターンごとに書かれるか → Stop フックは「未抽出の新規活動があるか」を判定せず、
  活動があったターンの終了で常に再装填する（抽出完了を知る術がない）。
- なぜそれが問題になるか → ゲート設計が「commit はセッション末尾に 1 回」という
  対話セッション前提で、複数ターン・複数 commit の自律ループ（pr-finalize-loop 等）では
  反復のたびにユーザー承認つき抽出が挿入され、スキルの自律性ポリシーと衝突する。
  ← [[2026-06-11-extractor-session-rearms-sentinel]]（適用済み）の「自己再装填」の新形態。
  旧件は撤去で解消したが、「フック自身の活動がセンチネルを再装填する」経路の横断確認が
  現行 Stop フックに及んでいなかった。

## 提案

自律ループ・通常対話セッションの双方で「ゲートはセッションにつき 1 回だけ抽出を要求する」
状態にする。抽出完了の事実をマーカーファイルで記録し、セッション境界で SessionStart フックが
マーカーを消す（**抽出完了マーカー方式**。session ID の stdin 解析はエージェントごとの入力
スキーマ差に依存するため採らない — ゲート自身が exit 2 方式を選んだのと同じ移植性判断）。

1. **抽出完了スクリプト（新規 `kaizen-extract-done.sh`）**: プロジェクトルート基準で
   `rm -f .kaizen/.pending-extract*` と `.kaizen/.extract-done`（UTC タイムスタンプ）の
   書き込みを 1 スクリプトで行う。extract.md 手順 8 とゲートの stderr 案内をこの呼び出しに
   差し替える（インライン redirect の cwd 相対問題 = 2026-06-16 の迷子センチネル対策も兼ねる）。
2. **ゲート（`kaizen-precommit-gate.sh`）**: センチネル検出後、`.kaizen/.extract-done` が
   あれば素通りさせる（同一セッション内で抽出済み。Stop フックの再装填を無効化する側）。
3. **SessionStart（`kaizen-context-inject.sh` 冒頭）**: `.kaizen/.extract-done` の削除を追加。
   新セッション開始でマーカーが消え、以降の活動で従来どおりゲートが効く。3 エージェントとも
   既に SessionStart 相当のフックとして設定済みのため追加設定は不要。
   ただし Claude Code の SessionStart は自動圧縮（source: compact）でも発火するため（この
   リポジトリの matcher は ""）、stdin の source が compact のときは削除しない（圧縮は同一
   セッションの継続。消すと長時間自律ループで摩擦が再発する）。source 不明時は削除側に倒す。
4. **Stop フック（`kaizen-stop-mark.sh`）は変更しない**（再装填は続くが、ゲートが
   マーカーで抑止する。センチネルの「未抽出活動あり」という意味は保たれる）。
5. `.gitignore` に `.kaizen/.extract-done` を追加する。

挙動: セッション内の最初の commit でゲート→抽出→マーカー記録、以降の commit は素通り。
次セッションの開始でマーカーが消えて再度 1 回だけゲートが効く。

既知の限界（許容）: (a) 同一リポジトリで複数セッションが並行すると、後発セッションの
SessionStart がマーカーを消し、先行セッションの次 commit が再ブロックされ得る（安全側）。
(b) SessionStart フックが発火しない環境ではマーカーが残留しゲートが開きっぱなしになる
（必要なら「マーカーは 24h で失効」を ゲートに足して残留を有界にできる）。
(c) 抽出後〜セッション末尾の新規失敗は当該セッションでは抽出されない（設計どおり。
次セッションのゲートと SessionStart 注入が拾う）。
