# 新側実装（現行コードを一次情報源に読む・確信度を申告する）

現行コードの読み方・照合単位の振り分け・確信度の常時申告・未解決の明示・振る舞い保存と品質改善の分離を定める。

## 現行コードは常にロジックの一次情報源

**現行コードはフロントエンド・バックエンドのいずれでも読む。** バリデーションルール・fetch・状態遷移・エラー処理・ログ送信は、コードを確認しないと再現できない。

非対称なのは「現行コードを読むか」ではなく**レビューの照合単位**である。

| | ロジックの情報源 | レビューの照合単位 |
|---|---|---|
| バックエンド（現行コードあり） | 現行コード | **旧新を並べた diff が成立する** |
| フロントエンド（1:1 対応が作れない） | 現行コード（ロジック）＋視覚ベースライン（見た目・構造） | **diff は成立しない。** パリティスイートが green か、`parity-diff` の差分がゼロか |

- `replace-strategy` の測定（現行コードの入手性）に従って振り分ける。現行コードが入手できない側はパリティスイートとベースラインだけが頼りになる
- **意図的差異レジストリの `keep`（テーブル名・項目名・API エンドポイント・同じ機能なら同じ関数名・スコープの広い変数名）が diff レビューを可能にしている。** ディレクトリ構成・ファイル名・狭いスコープの変数名は変えてよいが、変えるほど照合コストが上がる

## 推測せず、確信度を常に申告する

- **推測で実装しない。** 判断できない箇所は未解決であることをコード上に明示し（`TODO` コメント等）、レビューへ回す。**間違ったコードより未解決の明示のほうがよい**
- **確信度（高／中／低）を実装単位（ファイル、またはページ内のエリア）ごとに 1 行、常に申告する**（`porting.md`）
  - **迷ったときだけにしない。** 迷ったときだけの申告にすると「迷っていることに気づかなかった」箇所が素通りする。常に書けば「低」が普通の申告になり、言い出すコストが下がり、レビュー側の優先順位付けにも使える
  - **「低」は「おそらく間違っている。レビューで現行を読み直せ」の意味**とする
- **長い言い訳コメントが付いている箇所は、コードが間違っているサインとして扱う**

## 振る舞い保存と品質改善の分離

- **同じフェーズで両方を狙わない。** 忠実な移植は良い性質も悪い性質も等しく運ぶ前提で進め、レガシーの奇妙な挙動も再現する
- **リント起因の修正は許容する**（アクセシビリティ改善・不要な副作用の削除など）。**リントを off にしない**——ロケータマッピング層が現側の非セマンティックさを隔離しているため、新側を改善してもスイートは壊れない
- **改善によって role や構造が変わった場合は、意図的差異レジストリ（`intentional_diffs`）へ追記する。** `parity-diff` がそれをノイズフィルタとして使う
- **タブ順の厳密一致を目標にしない**（ARIA APG 準拠の実装で新の方が正しくてもタブ停止数が減りうる）
