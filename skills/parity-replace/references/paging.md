# ページ分割とフェーズ構成

機能をページ単位のフェーズに分けて 1 つずつ作り切る手順と、api-resource / batch モードの縮退を定める。

## ページ単位のフェーズ分割（feature モード）

機能 Issue は 1 度に作らず、**ページ単位でフェーズを切る**。

- **1 ページを作り切って比較してから次のページへ進む。** ページをまたいで並行に進めない（調査・実装・比較が浅くなり、差異を見落とす）
- ページの単位は `.replace/features.md` が機能に紐づけたページ一覧を起点にする。推測で足さない
- フェーズは「実装 → 新側マッピング充填 → 見た目の系統差の縮小 → 敵対的レビュー → スイート green ＋ 静的解析」までを 1 ページ分回し切る単位とする

## フェーズ内の順序（読み取り → 書き込み）

フェーズ内では**読み取り経路を先に、書き込み経路を後に**扱う。

- 読み取り経路は DB 復元が不要なため先に回すと安く比較できる
- これはフェーズ内の順序であって、機能の並び順の話ではない（機能の並び順は `replace-strategy` の依存順が決める）

## api-resource モード（横断 API リソース）

画面を持たない。**ページ分割・視覚差分・新側ロケータマッピングは行わない。**

- 担うのはバックエンド実装（現行コードとの diff レビュー）と、API パリティスイートを新に対して green にすることだけ
- 成果物パスの `slug` は当該リソースのもの
- 完了判定は API スイート green ＋ 静的解析（視覚 3 点セットは無い）。green 化の前に `golden-dataset --phase b` で新側 DB へ投入する

## batch モード（バッチ Issue）

画面系の工程を動かさない。

- バッチ本体の新側実装（現行コードとの diff レビュー）と、その出力（DB 状態・生成ファイル・対象範囲内の副作用出力）が現行と一致することを担う
- 出力一致の検証手段は `parity-suite` がバッチモードで採取した現行ベースライン（`.replace/parity/<slug>/` 配下）に対して行う
- 完了判定は出力一致 ＋ 静的解析。新側でバッチを走らせる前に `golden-dataset --phase b` で新側 DB へ投入する
