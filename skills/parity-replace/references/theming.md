# 見た目の系統差を源流で縮める（トークン寄せ・component_diffs 宣言・gap 記録）

新側 UI ライブラリを reference で選び、旧 design token を寄せ、テーマで消せない構造差を宣言または gap 記録する手順を定める。これは比較の正規化であって仕様変更ではない。

## 源流で縮める理由

コンポーネントライブラリが変わる以上、見た目はライブラリ由来で正当に違う。これを差分器側だけで説明しようとすると宣言が膨らむため、**実装側でも源流の差を縮める**。**論理名・振る舞い・構造は不変で、見た目の系統差だけを扱う。**

## 新側 UI ライブラリは reference で選ぶ（固定しない）

- 新側ライブラリと旧→新トークンマッピングは設定 `references.ui_library` が指すファイルから読む。**特定のライブラリ名・DB 名を本スキルやコードに固定で書かない**
- reference は `replace-strategy` の測定と `ui_library` 設定を基に生成・レビュー済みのプロジェクト知識

## 旧 design token を新側テーマへ寄せる

- **選ばれたライブラリがテーマ可能なら、旧アプリの design token（色・余白・角丸・タイポグラフィのスケール）を新側テーマに寄せる。** 旧→新マッピングは `ui_library` reference から与えられる
- **テーマ機能が無い／限定的なライブラリなら**、源流縮小の効きは限定的で、次項のクラス系統差宣言の比重が上がる

## テーマで消せない構造差の扱い

テーマで消せない構造差（フォーカスリング形状・内部 DOM・余白の配り方）は残る。二段構えで扱う。

1. **クラス/トークン単位で表せる系統差** → 共有契約の `component_diffs` レジストリ（`intentional_diffs` とは**別のキー**。要素の形の正本: `replace-strategy` の `references/project-config.md`）へ**ユーザー確認の上で宣言**する。`parity-diff` が比較の正規化に使う。
   確認が取れるまでは `component_diffs` にも `intentional_diffs.pending`（分類ラベルの列で要素の形が違う）にも書かず、対話で確認してから宣言する
2. **クラス/トークン単位でも表せない構造差**（フォーカスリング形状・内部 DOM・余白の配り方等） → `.replace/parity/<slug>/gaps.md` の「宣言できない構造差」節へ追記する（正本の様式は `parity-suite` の `assets/gaps-template.md`）。`parity-diff` の正規化対象外＝未検証として残す

いずれも比較の正規化であって仕様変更ではない。**論理名・振る舞い・構造は不変**に保つ。
