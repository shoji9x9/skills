# 構成図の作図ルール

AWS 構成図を「交差が少なく・軸が揃い・ラベルが読める」状態に保つための規約。
一般原則は公開資料に基づき、AWS 固有の文脈を補う。図はコード（spec）から生成し、
テキスト差分で管理する（GUI 不要）。**このルールが本スキルの中核**であり、spec を
書くとき・図を直すときは常にこの観点で確認する。

## 一般原則（公開出典）

- **エッジ交差の最小化を最優先**し、次いで屈曲（bend）削減・直交性・ノード非重なりを
  重視する（可読性への寄与が交差最小化で最大）。出典:
  [Metrics for Graph Drawing Aesthetics](https://www.researchgate.net/publication/222546202_Metrics_for_Graph_Drawing_Aesthetics)
- **直交配線（orthogonal routing）**: エッジは水平/垂直線分のみで、ノード/ラベルを避けて
  通す。複数の線が同じチャネルを共有する場合は一定量ずらして重なりを防ぐ。出典:
  [yWorks – Drawing Orthogonal Diagrams](https://www.yworks.com/pages/drawing-orthogonal-diagrams)
- **矢印は流れの向き**を表し、性質の異なる線（認証・非同期）は破線で区別する。出典:
  [C4 model – Notation](https://c4model.com/diagrams/notation)
- **AWS 公式アイコン**はラベル必須・直線優先・余白確保・バージョン混在禁止。出典:
  [AWS Architecture Icons](https://aws.amazon.com/architecture/icons/) /
  [AWS diagram guidelines](https://dev.to/aws-builders/aws-architecture-diagrams-guidelines-595d)

## 本スキル固有のルール

spec を書くときに守る具体規約。starter kit の描画エンジン（`diagram-engine.mjs`）は
一部を機械的に強制する（重複エッジ検査・アイコンパス制限・交差の飛び越し自動付与）。

- **グループ枠（`groups`）を相互に重ねない。** 各ノードは所属グループ枠の内側に
  アイコン半分（32px）以上のマージンを取って収める。
- **ノードは軸方向に揃える。** 同じ層のノードは x または y を共有して格子に乗せる
  （例: 永続層は x を 1 つに揃えて縦一列）。
- **依存先グループは、アプリ層からの線が無関係なグループを貫通しない位置に置く。**
  例えば外部 API を永続層の**上段**に積み、アプリ層 → 外部 API の線が永続層を跨がない
  ようにする。
- **エッジは直交。** `waypoints` は前後の点と x か y のいずれかを共有させ斜め線を
  作らない。直線で結べるものは `waypoints` を付けず自動 L 字に任せ、屈曲は他ノード/
  ラベル回避・交差回避の目的がある時だけ足す。
- **ノード間の矢印は最大 1 本。** 同じ `from`→`to` を重複させない（`diagram-engine.mjs`
  の重複チェックが違反時にエラーを投げる）。
- **1 つの辺に複数の線が入ってよいが、進入位置をずらして区別する。** 複数の依存元が
  同じ先に接続する場合、入る辺を変えるか、同じ辺でも進入 y/x をずらす（アンカーは
  最後の `waypoint` 座標に沿って辺上の位置を決めるので、`waypoint` で進入位置を制御
  する）。1 点に集約すると「どの線がどこから来たか」が分からなくなる。
- **多数の線が集まる領域は共有の「幹線」x に一旦寄せてから分岐**させ、平行線は最低
  20px 離す。幹線で無関係なアイコン（中心 ±32px）を貫通させない。
- **交差は飛び越し（line jump）で示す。** まず配置・進入位置で交差を減らし、残った交差は
  水平線側に小さな半円アーチ（∩、上方向に膨らむ）で「交差＝非接続」を明示する
  （`diagram-engine.mjs` が自動付与。水平線が他エッジの垂直線を飛び越す）。
- **線はグループ枠線とも重ねない（枠線に沿わせない）。** 枠を跨ぐのは直交の通過のみと
  し、枠線と同じ x/y に平行な線を置かない。
- **キャンバス端に余白を取る。** 最外のグループ枠がキャンバス端に接しないよう `W` / `H`
  に余白（おおむね 50px 以上）を持たせる。
- **ラベル（`lp`）は線の出ていない辺へ寄せる**（top / bottom / left / right）。ラベルを
  置いた辺に線を引いたら `lp` を別の空き辺へ移す。
- **アイコンは AWS 公式（`aws-icons/`）と非 AWS（`browser` / `internet` 等）を使い分ける。**
  取得・出典は [icons.md](icons.md) を参照。バージョンを混在させない。

## 生成と品質確認

spec を編集 → 図を再生成 → PNG にラスタライズして目視確認 → 座標・`waypoints` を
微調整、を反復する。**SVG は画像参照ツールで直接描画できないため、必ず PNG 化を挟む。**

```bash
cd docs/diagrams   # 図ディレクトリ（$SKILL は本スキルの導入先）

# 1. SVG を生成（既定=全環境。環境選択は environments.md を参照）
node "$SKILL/assets/engine/render-diagram.mjs"

# 2. PNG にラスタライズ（出力パスが表示される）
node "$SKILL/assets/engine/preview-diagram.mjs" prod
node "$SKILL/assets/engine/preview-diagram.mjs" local
```

出力された PNG を**画像として読み**、次の観点で確認する。1 つでも崩れていたら座標や
`waypoints` を直して再生成する。

- **(a)** グループ枠・ノードが重なっていない
- **(b)** 線が斜めでない（直交）／無関係アイコンを貫通していない
- **(c)** 平行に走る線が重なっていない（最低 20px 離れている）
- **(d)** ラベルを線が貫通していない
- **(e)** 矢印の向きが流れ（`from`→`to`）と一致している
- **(f)** ノード間の矢印が最大 1 本で、同じ辺への複数進入が位置で区別できている
- **(g)** アプリ層からの線が無関係なグループを貫通していない
- **(h)** 線がグループ枠線と重なって（沿って）いない
- **(i)** 最外の枠とキャンバス端に余白がある
- **(j)** 交差点に飛び越し（line jump）が付き、接続と区別できている

## 座標の決め方（実務のコツ）

- **層ごとに基準線を決める。** アプリ層は y を数本（例 380 / 540）に、永続層は x を 1 本に
  揃えると、格子に乗って交差が減る。
- **幹線を先に引く。** 多対多で集まる領域（例: アプリ層 → 永続層）は、まず縦の幹線 x を
  2 本（依存元ごと）決め、そこから各ノードへ分岐する。幹線同士は 20px 以上離す。
- **進入 y をずらす。** 同じノードの同じ辺に複数入るときは、最後の `waypoint` の y を
  少しずつ変えて辺上の進入点を分散させる。
- **外向きの線は層の外周から出す。** 外部 API へ向かう線は、無関係グループの上/下の
  余白（グループ間の隙間）を通してから目的地へ寄せる。
