# 認証・認可

**原則として認証が必要な前提で設計する。** UI・API のいずれも、未認証では特性化できない。

## 認証情報の扱い

シークレットの規律（環境変数から読む・設定には変数名のみ・`secrets.wrapper` の前置・値を出力/復唱しない）は
**`replace-strategy` の `references/project-config.md`「シークレットの扱い」を正本**とし、ここでは再定義しない。本スキル固有の点だけを定める:

- 読むキーは `auth.current.env_vars` / `auth.new.env_vars`。現・新それぞれの認証方式と認証情報を持つ（方式が異なりうるため側ごとに分ける）。本スキルは `current` を使う
- **値をスクリーンショット・スナップショット・ネットワークログ等の採取成果物にも含めない**（マスキングは下記）

## UI の認証

- **ログインを毎テストで繰り返さず、Playwright の `storageState` で認証状態を再利用する**
- **ロールごとに `storageState` を分ける**（下記の認可のため）

## API の認証

- **トークンの取得と更新を仕組みとして持つ。** 有効期限切れでテストが落ちる状態にしない

## 認可はそれ自体が仕様

誰が何を参照・作成・更新できるかは仕様である。

- **ロール・所属・関与条件ごとに代表ユーザーを用意する**
- 権限で表示・操作・応答が変わる箇所は**各ロールで特性化する**
- **401 と 403 の使い分けも対象**（認証されていない／認証済みだが権限が無い、の区別）
- 認証が必要な範囲を特定できない場合は「未検証」として `gaps.md` に残す（確認済みにしない）

## マスキング

- スクリーンショット・スナップショット・ネットワークログに**認証情報・トークン・個人情報を残さない**。動的領域のマスク方針は [`baseline.md`](baseline.md) の撮影条件に従う
