# バージョン運用・陳腐化・parity-suite との往復

データセットの `version` は、`parity-suite` / `parity-diff` の成果物が「どのデータで取得したか」を追跡し、土台のデータが変わったベースラインを陳腐化として検出するための鍵である。

## version のセマンティクス

`metadata.json` の `version` は **1 始まりの単調増加の整数**。

- **上げるのは論理データが変わったときだけ**（行の追加・変更、エッジケースの追加、`base_time` の変更）。1 回の変更で +1
- **フェーズ B では上げない**。フェーズ B は論理データを新側スキーマへ写像するだけで、論理データ自体は変えないため
- `base_time` の変更は論理データの変更として扱う（時刻依存データが全部変わるため → +1）

## 陳腐化検出の全体像

`version` を各下流成果物が記録し、「記録 < 現在」で陳腐化を検出する。

| 読み手 | 記録する場所 | 検出 |
|---|---|---|
| `parity-suite` | `.replace/parity/<slug>/metadata.json` の `dataset_version` | ベースライン採取時の version。現在より古ければ再取得を宣言 |
| `parity-diff` | 同上を参照 | 陳腐化したベースラインでの比較を避ける |
| `replace-strategy status` | 上記を集約 | `dataset_version` < `.replace/dataset/metadata.json` の `version` なら「要再取得」と報告 |

**データセットを刷新すると、それ以前に取得した視覚ベースラインと特性化結果は土台のデータが変わった時点で無効になる。** version はこの無効化を機械的に検出させる。

## parity-suite の gaps.md「データ不足」からの戻りループ

`parity-suite` が現行アプリを探索する過程で、シードに足りないケースが必ず見つかる（このパターンのデータが無いと空リストしか確認できない、ページネーションが 1 ページで終わる、等）。これは `gaps.md` に「データ不足」として記録され、本スキルへ戻る入口になる。

戻ってきたら（フェーズ A 再実行）:

1. **設計追記**: `design.md` に不足パターンを追加する（何を追加したかを「gaps 由来の追加履歴」に version 付きで残す）
2. **ツール更新・再投入**: 追加分を含めて投入ツールを更新し、現行テスト環境へ再投入する（冪等なので全体を入れ直す）
3. **再検証**: FK 整合・必須項目・件数・カバレッジを再確認する
4. **`version` を +1**: `metadata.json` を更新する
5. **影響ベースラインの再取得を案内**: この version 未満で取得された `parity-suite` のベースラインは陳腐化する。**`parity-suite` の再実行を案内する**

このループは一度きりではなく、探索が進むたびに繰り返される前提で運用する。
