# コンポーネント選択基準

追加したい知識・規約・自動処理を、skill / rule / hook / ドキュメントのどれに落とすかを判断する基準。
スキルを明示的に呼んだ場合だけでなく、**ドキュメントの整理依頼やエージェントの自主的なドキュメント作成・分割の場面でも適用する**。

## 判断の入口

追加したいものの性質で振り分ける。

| 追加したいもの | コンポーネント |
|-------------|-------------|
| 人／エージェントが実行する一連の手順・ワークフロー | スキル（`references/skills.md`） |
| 特定のファイル群を触るときに守らせたい規約 | ルール（`references/rules.md`。paths で自動適用） |
| 特定イベントで自動実行したい決定論的な処理（ゲート・整形・検査） | Hook（`references/hooks.md`） |
| 上記に当たらない知識・方針・仕様 | ドキュメント（`references/docs.md`） |

## ルール vs ドキュメントの分岐

規約・方針を残したいとき、ルールとドキュメントの境界は「いつ関係するか」で決める。

- **特定のファイル群を触るときだけ関係する規約** → ルール。paths で対象を絞り、その範囲を触るときだけ自動適用させる。
- **作業対象に依らず常に把握が必要な方針・規約** → ドキュメント（基底ドキュメント。後述）。

ルールを選んだら paths は最小スコープで切る（`references/rules.md` の「rules 化とスコープの基準」を参照）。

## ドキュメントの中での置き場所

ドキュメントに落とすと決めたら、置き場所を選ぶ。

- **全エージェント共通で常に把握が必要** → 基底ドキュメント（下記）。single source of truth をここに置き、他は複製せず参照する。
- **特定スキルの実行時にだけ要る詳細手順** → そのスキルの `SKILL.md` / `references/`。
- **長大な参照情報・設計資料** → `docs/` 等の専用ドキュメント。基底ドキュメントからは参照だけ張る。

### 基底ドキュメントとは

プロジェクトが基底として常時ロードする指示ドキュメント。構成によって実体が異なる。

- マルチエージェント構成: `AGENTS.md`（`CLAUDE.md` / `.github/copilot-instructions.md` は差分のみ）
- Claude Code のみ利用: `CLAUDE.md`
- GitHub Copilot のみ利用: `.github/copilot-instructions.md`

`AGENTS.md` を持たない構成では、実際に使っている常時ロード doc（`CLAUDE.md` か `.github/copilot-instructions.md`）が基底ドキュメントの役割を担う。「AGENTS.md へ」と書かれた箇所はその基底ドキュメントに読み替える。

## この基準自体の置き場所

この選択基準は「ドキュメント・規約を追加／整理する」たびに関係するため、特定ファイルを触るときだけ関係するわけではない。
したがって基準自体は **paths 付きルールにしない**（ドキュメント群を paths で綺麗に捕捉できず、`**/*.md` では広すぎる問題に逆戻りする）。
代わりに、基底ドキュメントへ短い判断基準（この表と分岐の要点）を置き、詳細は本ファイルを参照させる。これにより、スキルを明示的に呼ばない場面でも常時適用される。
