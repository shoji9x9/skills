# 強度の検証（故障注入＝ネガティブコントロール）

**この工程が抜けると下流が全部無意味になる。** assertion が弱められた／そもそも無いテストも pass するため、テストがあること自体は品質の証拠にならない。

## 位置づけと実行タイミング

- **authoring 時のゲート**。以後はスケジュールではなく、**スイートの assertion が変わるたび**（穴埋め・データセット更新でベースライン再取得した等）に再実行する
- **強度は機能ごとに `strength.md` に記録し、`replace-strategy status` が参照する**

## ブラックボックス前提

古典的な mutation testing は SUT のソースを変異させる前提だが、本スキルには字義どおり乗らない。パリティスイートは現行アプリの外側から当てるブラックボックスであり、**現行を変更しない**（変更しないことが設計原則。加えて現行はソース変異と mutant ごとの再デプロイが非現実的な生成物でありうる）。

そこで**変異対象を差し替え、テストが観測する境界に既知の故障を注入する（ネガティブコントロール）**。

## 判定単位は「現行検出一式」

**強度は「手書き assertion 単体」ではなく「現行検出一式＝手書き assertion ＋ ベースライン ＋ 差分器」で判定する。** 故障を注入して、この一式のどれもが赤くならなければ、その故障種別に対して一式は弱い（例: ボタン色を変えて手書き assertion も差分器も赤にならなければ、一式はスタイルに対して弱い）。

## 故障カタログ

**恣意的に作らず、既知の回帰分類から導出する**——意図的差異レジストリ、`references.db_semantics` の差、コンポーネントライブラリ差、既知の移行ハザード。故障クラスと注入手段:

| 故障クラス | 例 | 注入手段 | 拾う経路 |
|---|---|---|---|
| 値・件数・並び（データ由来） | 応答の値・順序の改変 | **上流摂動**（`page.route` で応答を摂動し描画経路へ伝播） | 手書き assertion（＋差分器） |
| 構造（aria） | role・構造の欠落 | DOM 摂動（`page.evaluate`） | 手書き assertion ＋ aria 比較（摂動後に aria を再取得し**ベースライン相手に**構造比較） |
| 静的ラベル・placeholder・role | 文言・placeholder の改変 | **DOM 摂動**（`page.evaluate`。上流で届かない静的マークアップ用の第二経路） | 手書き assertion |
| 要素スタイル（色・余白・フォント・罫線） | 色・余白等の改変 | **スタイル注入**（`page.evaluate` / `addStyleTag` / CSS 応答摂動）して撮り直す | 差分器を**ベースライン相手に**回す（新側不要） |

- **検出＝手書き assertion が赤 ∪ 差分器（特性照合／画素／aria 比較）が赤。** 名前付き要素のスタイルは特性照合が、名前無しは画素が拾う。
  aria 比較＝取得済み aria スナップショット（参考資料）のベースラインとの構造比較で、`parity-diff` の第 3 経路（テーブル/フォームの内容パリティ）に相当する
- **両方を素通りした故障が一式の弱点**——強化するか、`gaps.md` に記録する（確認済みにしない）

## 差分器の健全性の確認

強度ゲートでは差分器の健全性も確認する。対象は `parity-diff` が再利用する **3 経路すべて**（特性照合・画素・aria 比較）:

- 注入した**既知の差を差分器が赤にできる**こと（スタイル注入 → 特性照合・画素、構造の DOM 摂動 → aria 比較）
- **注入量がノイズ基準値に埋もれない**こと（ノイズ基準値は [`baseline.md`](baseline.md)）

**ここで検証した差分器を `parity-diff` が現新比較にそのまま再利用する**（ツール・しきい値・ノイズ基準値は `metadata.json` の `differ` / `noise_baseline` 経由で引き渡す）。

## カタログの外は射程外

故障注入が与えるのは強度の**下限**（「少なくともこれらは捕捉した」）であって完全性の証明ではない。緑は「反例が見つからなかった」に過ぎない。**捕捉した故障種別と未検証の種別を `strength.md` に記録し、「強い」と宣言しない。**

- **新側 TypeScript・自作コード（マッピング層・データセット生成ツール）の本物の mutation testing は本スキルのスコープ外**（通常の開発衛生であり、パリティの強度検証とは別物）
