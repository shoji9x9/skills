# 学び適用ガイド

## フロー概要

`.kaizen/` 内の `status: pending` ファイルを根本原因・種別でグルーピングし、グループ単位で 1 つの対策を作成する。`multiagent-setup` スキルを活用して成果物を作成する。

## 手順

### 1. 学びファイルの一覧とグルーピング

```bash
ls .kaizen/*.md 2>/dev/null
```

`status: pending` のファイルがなければ「適用待ちの学びはありません」とユーザーに伝えて終了する。

ファイルを Read して内容を把握し、以下の観点でグルーピングする:

- 同じ根本原因を持つファイルをまとめる
- `type` が同じで対象ファイルや対策が共通するものをまとめる
- 明らかに無関係なものは別グループとする

グループ一覧をユーザーに提示し、グルーピングが適切か確認する。

> 本ファイルの各確認・承認では、判断材料（グループ一覧・統合した事象・根本原因・提案・成果物のドラフト）を確認 UI 自体に同梱する。直前の通常テキストが確認ダイアログと同時に見えることを前提にしない（Claude Code の AskUserQuestion では選択肢の preview フィールドに入れる）。

### 2. グループごとに適用（繰り返し）

各グループについて:

1. グループ内のファイル一覧と統合した事象・根本原因・提案をユーザーに提示する
2. 適用するか確認する（承認 / 却下 / スキップ）
3. **承認された場合**: Step 3 へ進む
4. **却下された場合**: グループ内の全ファイルを `status: rejected` に更新して次へ
5. **スキップの場合**: そのまま次へ

### 3. 成果物の作成

`type` フィールドに応じて `multiagent-setup` スキルを使用して成果物を 1 つ作成する。

| type | 適用先 | multiagent-setup コンポーネント |
|------|--------|-------------------------------|
| `rule` | `.agents/rules/<name>.md`（`paths`/`applyTo` で対象を絞れる場合。`**` のみなら rule の利点が薄いので `AGENTS.md` / hook を検討） | rule |
| `skill` | `.agents/skills/<name>/` | skill |
| `hook` | `.claude/settings.json` 等 | hooks |
| `doc` | `AGENTS.md` / `CLAUDE.md` 等 | docs |
| `other` | ユーザーと相談して決める | — |

#### 記述先（適用先）の選び方

`type` は出発点に過ぎない。**どこに書けばエージェントが期待する挙動を取る確率が最も高いか**を、以下の観点で検討して適用先を決める。

- **決定性で選ぶ**: 散文の指示（rule / doc）はエージェントが従わない確率が残る。確定的に効かせたい再発防止は、リンター・pre-commit・Hook など**機構（hook / script）を優先**する（基本原則「仕組みで解決する」）。
- **責務の所在で選ぶ**: 特定エージェント固有の特性は当該エージェントの設定（例: `CLAUDE.md`）、全エージェントに共通する一般原則はプロジェクトの `AGENTS.md`、特定スキルの運用に閉じる知見はそのスキル内に置く。
- **パススコープが効くかで選ぶ**: 特定のパス配下での作業時だけ必要なルールは `paths` / `applyTo` を絞った rule にすると、該当ファイル編集時に確定的にロードされる（例: スキル作成時のみ効かせたいなら `paths: skills/**`）。全ファイル対象（`**`）にしかできないものは rule の利点が薄いので doc / hook を検討する。
- **配布スキルに関わる知見は同梱して運ぶ**: 他プロジェクトへ配布・インストールして使うスキルの、使い方や規約に関わる知見は、
  それが置かれたリポジトリのドキュメントにだけ書くと、スキルを単体配布したときインストール先へ伝わらない。スキル本体
  （`SKILL.md` / コンポーネント / `assets` 等）に同梱して運ぶのを基本とする（スキルを使うエージェントが必ず読むため、
  スキル固有の挙動はこれが最も確実）。インストール先でも確定的に効かせたい再発防止は、ドキュメントへ散文を追記させるのでは
  なく、スキルのインストール手順で hook / lint などの機構を導入する（「決定性で選ぶ」参照。散文の追記は守られない確率が
  残るため最後の手段）。

迷う場合や `type` と最適な適用先がずれる場合は、上記を踏まえた候補をユーザーに提示して決める。

グループ内に複数ファイルがある場合、それら全ての内容を統合して 1 つの成果物に反映する。

`type: rule` の場合: 各ファイルの「提案」セクションを統合してルール文面を作成する。

### 4. 成果物の確認と後処理（status 更新は必須）

1. 作成した成果物をユーザーに確認する
2. 問題なければグループ内の**全ファイル**の `status` を `pending → applied` に更新する（**反映したら必ず実施。ここを飛ばすと反映済みなのに `pending` のまま残る**）
3. 修正が必要な場合は修正後に更新する
4. グループ処理の直後に、反映済みのファイルに `pending` が残っていないか確認する:

```bash
# 反映済みグループのファイルが applied になっているか確認する
grep -l "^status: pending" .kaizen/*.md 2>/dev/null
```

### 5. 全グループ処理後の対応

すべてのグループを処理した後、まず status の取りこぼしが無いか最終確認する。反映したのに `pending` が残っているファイルがあれば `applied` に是正する（反映していないものは `pending` のままで正しい）:

```bash
grep -l "^status: pending" .kaizen/*.md 2>/dev/null
```

続いて、適用済みグループについてユーザーに確認する:

- **このブランチで作業継続**: コミットして次の作業へ進む
- **Issue を作成**: グループ（= 対策）ごとに 1 Issue を作成し、別ブランチで詳細を詰める

## クリーンアップ（整理）

`.kaizen/` のファイルが増えすぎた場合は、適用済み・却下済みの学びを整理する。手順は `references/housekeeping.md` を参照する。既定は `.kaizen/archive/` への**アーカイブ（非破壊な移動）**で、物理削除はユーザーが明示したときだけ行う（対象は対話で確認する）。

## `.kaizen/` の Git 管理

`.kaizen/` ディレクトリはコミット対象とする（`.gitignore` に追加しない）。`.kaizen/archive/`（整理で移動した applied / rejected の置き場）も同様にコミット対象とする。

- `status: applied` / `status: rejected` の履歴が残り、改善の経緯を追跡できる
- チーム内で学びを共有・レビューできる
- `.kaizen/` 内の全ファイル（ステータス問わず・`archive/` 配下を含む）はどのエージェントも参照できる（Claude Code / Codex / Copilot 間の学び共有）
