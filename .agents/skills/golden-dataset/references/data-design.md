# データ設計（フェーズ A）

一から作る以上、**何を含めるかが品質のすべて**になる。データ設計は現行と新側の値比較を成立させる基盤であり、`.replace/dataset/design.md` に記録する（正本テンプレート: [`../assets/design-template.md`](../assets/design-template.md)）。

## 起点は DDL と機能インベントリ

- **DDL の制約**: テーブル、必須項目、型、一意制約、外部キーを起点にする。各テーブルの必須・一意・FK を満たす最小構成をまず作る
- **機能インベントリ**（`.replace/features.md`）: どのテーブルが対象か、どのリソースドメインに属するかを引く。対象テーブルは features.md の記録に従う（推測で足さない）

## エッジケースのチェックリスト

表示・挙動の比較で差が出る箇所を**意図的に**含める。各項目について「含めた／含めなかった（理由）」を design.md に残す。

- null（許容カラムに実際に null が入る行）
- 境界値（数値・日付の下限／上限、0・空文字と null の区別）
- 最大長（カラム定義の最大長ちょうどの値。表示の折り返し・省略に影響する）
- 権限・ロールのパターン（各ロールが少なくとも 1 件）
- 状態遷移の各状態（取りうる全ステータスをそれぞれ 1 件以上）
- **空リストになるケース**（フィルタ結果が 0 件になる条件を作れるデータ配置）
- **ページネーションが 2 ページ以上になる量**（1 ページで終わらない件数）
- ソート順が判別できる並び（同点で崩れない、順序が視認できる値の散らばり）

## 現実的な値の規律

**表示の検証に影響する項目に注意する。** 文字幅・桁数・改行の有無はレイアウト比較に影響する。

- 現実的な長さと文字種にする（`テスト1` のような値ばかりにしない）
- 全角・半角混在、改行を含む値、桁数の大きい数値など、実運用で現れる形を代表させる
- ただし非現実的な値ばかりにもしない（禁止事項 6）。**現実的な分布の中にエッジを混ぜる**

## 未カバー領域の記録

**「代表性がある」ことは証明できない。** 何を含め、何を含めなかったかを記録し、**未カバーの領域を明示する**（確認済みにしない）。

- design.md の「未カバーの領域と理由」節に、含めなかったパターンとその理由を残す
- `parity-suite` の探索で不足が判明したら追加する前提で作る（→ フェーズ A 再実行。[`versioning.md`](versioning.md)）

## 基準時刻の決め方

時刻に依存するデータは**基準時刻（`base_time`）を固定**し、そこからの相対で生成する。

- 「今週」「先週」の判定など、**週境界・日境界のロジックを持つ機能では特に重要**。境界の内・外に落ちる相対時刻を意図的に配置する
- `base_time` は `metadata.json` に記録し、投入ツールはハードコードせずここから相対時刻を計算する
- **基準時刻の変更はバージョンアップとして扱う**（論理データが変わるため。[`versioning.md`](versioning.md)）
