# 敵対的レビュー（ローカルの未コミット差分・実装役とレビュー役の分離）

commit 前にローカル差分へ敵対的レビューをかける具体手順と、`review.md` への記録を定める。

## 原則

- **ローカルの未コミット差分に対して実施する。** PR を作ってからではなく、`issue-start` の commit へ渡す前に回す
- **実装役とレビュー役を分離する。** レビュー役には**差分のみ**を渡し、実装の意図を知らせない
- **レビュー役の仕事は動かない理由を見つけること**であって、理解を示すことではない
- **「型検査が通った」「テストが通った」を理由にレビューを省略しない**

## 手順

1. **静的解析を先に通す**: 差分取得の前に静的解析（設定 `static_analysis`）を通し、自明な破綻を安価に落とす（通ったことをレビュー省略の理由にしない）
2. **差分の取得**: `git diff`（未ステージ）と `git diff --staged`（ステージ済み）でローカルの未コミット差分を取得する。範囲を `review.md` の「レビュー対象」に記録する
3. **レビュー役へ差分のみを渡す**: 実装役とは別のサブエージェント（Agent ツール）をレビュー役として起動し、**差分のテキストだけ**を渡す。実装意図・確信度・設計判断を渡さない（バイアスを持ち込ませないため）
4. **レビュー役の観点**: 現行コード（一次情報源）に照らして動かない理由を探す——ロジックの取りこぼし、状態遷移・エラー処理・バリデーションの差、`porting.md` の `TODO`（未解決）に対応するコードが本当に未解決のままか、`keep` の項目（テーブル名・API・関数名）を勝手に変えていないか
5. **指摘 → 修正 → 再レビュー**: 指摘があれば実装役が修正し、再度レビュー役へ差分を渡す。指摘が尽きるまでラウンドを回す
6. **記録**: 各ラウンドの指摘と対応、最終判定を `review.md` に残す

## `review.md` の位置づけ

- 敵対的レビューは **PR のレビュー機能（PR コメント・レビュースレッド）上では行わず**、ローカルの未コミット差分に対して実施する。その記録が `.replace/parity/<slug>/review.md`
- 記録ファイル自体は他のテキスト成果物と同様 **Git 管理してよい**（レビューを PR 上で行わないことと矛盾しない）
- 正本テンプレートは `assets/review-template.md`
