# 測定手順

`setup` モードのステップ 3。**すべて実測する。推測は「未測定」と明記する**（測れない場合の扱いは末尾「停止条件」を参照）。

結果は `.replace/survey.md`（テンプレート: [`../assets/survey-template.md`](../assets/survey-template.md)）に記録する。各測定値には測定方法・対象・日時を添え、後から再測定・比較できるようにする。

## 1. セマンティクス測定（role プローブ）

現行アプリの代表ページに role プローブを流し、**role ＋アクセシブルネームで引ける要素の割合**を出す。この値で、ロケータマッピング層の規模（`parity-suite`）と、HTML の id/name を変更してよいか（意図的差異レジストリの `pending` を確定）が決まる。

手順:

1. chrome-devtools MCP で現行アプリの対象ページを開く（認証が必要ならユーザーにログインを依頼する）
2. 同梱の [`../scripts/role-probe.mjs`](../scripts/role-probe.mjs) を読み、`roleProbe` 関数のリテラルを chrome-devtools MCP の `evaluate_script` にそのまま渡して実行する
3. 返り値（総数・named／unnamed・role 別内訳・(role, name) の重複数・unnamed のサンプル）をページごとに記録する
4. 対象ページは機能インベントリの代表ページから選ぶ。**最低でも一覧系・フォーム系・ダイアログを含む 3 ページ以上**。全ページでなくてよいが、対象にしたページを明記する

注意:

- プローブのアクセシブルネーム算出は**仕様の近似**（アクセシビリティツリーの完全な実装ではない）。値は「引ける割合の目安」であり、`parity-suite` 構築時に個別ページで再確認される前提で扱う
- **(role, name) が重複する要素はそのままではロケータとして一意に引けない**。重複数もマッピング層の規模見積もりに含める
- 測定値が低い（引ける要素が少ない）ことは停止理由ではない。ロケータマッピング層が厚くなるという**戦略への入力**である。停止するのは**測定自体ができない**とき（MCP が使えない、現行アプリに到達できない等）

## 2. DB 復元可否

スナップショット／リストアができるかを確認する。**手段は問わない能力の測定**であり、特定サービスを前提にしない（DB ネイティブのスナップショット・dump/restore・コンテナのリセット・マネージド DB のスナップショット等）。この結果で書き込み側パリティスイートの強度が決まる。

- 実際に「スナップショット取得 → 無害な変更 → リストア → 変更が消えたことを確認」まで通して初めて「復元可」とする。手段があるはずだ、では「未測定」
- 対象は**テスト環境の DB**。本番 DB には触れない
- 接続は環境変数（`skills.replace-strategy.current.db.env_vars`）＋起動ラッパー経由。値を出力しない
- 復元にかかる時間も記録する（書き込みテストの実行コストの見積もりに使う）

## 3. 現行コードの入手性

バックエンド／フロントエンドのコードが揃っているかを確認する。この結果で**バックエンドは直接移植、フロントエンドは別戦略**という非対称設計の可否が決まる。

- `current.repo` のコードが実際に読めるか、ビルド生成物ではなくソースか、フロントエンド／バックエンドそれぞれについて確認する
- 部分的にしか無い場合（例: バックエンドのみ）はその旨を記録し、無い側はパリティスイートとベースラインだけが頼りになることを戦略に反映する

## 4. 副作用の棚卸し

メール送信・外部連携・非冪等な操作を洗い出す。コードが読めるならコードから（送信・HTTP クライアント・ジョブ登録の呼び出し箇所を grep）、読めないなら画面と設定情報から推定し、**推定であることを明記する**。

- 各副作用に、パリティ検証の対象か対象外かを付ける。判定基準は**現行アプリを変更せずに現・新の結果を比較できるか**:
  - ファイル出力（CSV/Excel 等）・帳票/PDF: **対象**（生成物を read-only で捕捉して比較できる）
  - メール送信・外部連携: **スコープ外**（捕捉に現行アプリの変更が要る）。`gaps` に「手動検証が必要」として記録する（確認済みにしない）
- 副作用は独立した Issue にせず、**それを生成する所有者（機能ページまたはバッチ）の一機能として**インベントリに記録する

## 5. 既存テストの有無・密度・実装非依存性

現行のテストが新旧両方に当てられるか（実装言語・フレームワークに依存していないか）を確認する。

- HTTP・DOM 経由の E2E テストは新旧共通の判定基準として再利用できる可能性がある。ユニットテストは実装に結合しているため対象外
- 件数・対象範囲・実行できるか（実際に 1 回流す）を記録する

## 停止条件

次のいずれかに該当したら、**戦略提示へ進まず停止**し、不足と解消手順をユーザーに示す:

- chrome-devtools MCP が使えない（セマンティクス測定ができない）
- 現行アプリの対象環境に到達できない（URL 不明・疎通不可・ログイン不可）
- DB 接続の環境変数が用意されていない（存在確認で失敗。値は表示しない）

DB 復元「不可」やコード「無し」は停止条件ではなく戦略への入力である（復元不可 → 書き込み側は自己作成データ＋後始末へフォールバック、コード無し → 当該側はスイート＋ベースライン駆動）。
