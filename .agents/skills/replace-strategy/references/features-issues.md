# 機能インベントリと Issue 起票

`setup` モードのステップ 7（インベントリ作成）と `issues` モード（起票）の詳細。インベントリの正本は `.replace/features.md`（テンプレート: [`../assets/features-template.md`](../assets/features-template.md)）。

## 機能インベントリ（`.replace/features.md`）

現アプリを機能単位に分解し、次を記録する。

- **機能一覧**: 各機能が持つページ・API・テーブル・副作用出力（所有者として記録。対象／スコープ外の別も）
- **依存順**: 「依存の浅い順」とは具体的に、**横断 API の Issue を機能 Issue より先に**、という意味になる。ゴールデンデータセット Issue は全 Issue の前提として先頭
- **横断 API**: API 単位で**利用側の機能一覧（fan-out）**を持ち、2 つ以上の機能から使われる API に「横断」の印を付け、**リソース単位でグルーピング**する
- **バッチ**: バッチ・スケジュール実行の単位（画面を持たない）。入力・出力（DB 状態・生成ファイル）を記録する
- **slug**: 下記規則で採番し記録する
- **Issue 化の状態**: 起票済みの Issue 番号（`issues` モードが書き戻す）

### slug の採番（成果物パスの識別子）

- 各機能・各横断 API リソース・各バッチに、**ASCII の kebab-case（`[a-z0-9-]`）で一意な `slug`** を付ける
- 日本語の機能名は転写せず、作業内容を表す短い英語に要約する（例: 注文一覧＋注文詳細 → `order`）
- 下流スキルの成果物パス `.replace/parity/<slug>/` はこの slug を使う。**下流スキルは slug を自分で作らず、features.md から引く**（別々に採番すると成果物が合流しない）
- 一意性はインベントリ全体（機能・横断 API リソース・バッチの間）で保つ

### 横断 API のグルーピング

- **リソース単位で Issue にまとめる**: 同じリソースを扱う横断 API は 1 つの Issue にする（例: ユーザー一覧取得とユーザー詳細取得は「ユーザー」の Issue）。**全横断 API を 1 Issue にまとめない。API ごとに 1 Issue も作らない。粒度の基準はリソース**
- 機能に閉じる API（1 機能からしか呼ばれない）は機能 Issue に含める
- 横断 API はバックエンドのみで、パリティスイートの作り方も画面駆動ではなく API の record/replay になり、機能 Issue とワークフローが異なる（`parity-suite` / `parity-replace` の API モード）

## Issue の種類（4 つ）

| 種類 | 単位 | 依存順 |
|---|---|---|
| ゴールデンデータセット Issue（`golden-dataset`） | 全機能横断で 1 本（規模が大きければリソースドメインで分割） | **先頭**（他のすべての Issue の前提） |
| 横断 API Issue | リソース単位 | 機能 Issue より先 |
| 機能 Issue | 機能単位。バックエンドとフロントエンドの両方を含む | 横断 API の後、依存の浅い順 |
| バッチ Issue | バッチ・スケジュール実行の単位 | 入力データと参照テーブルが揃った後 |

### 本文に含める内容（共通）

- 対象の slug と `.replace/features.md` への参照（正本はインベントリ。本文へ全転記しない）
- 使用する姉妹スキルと順序
- 受け入れ条件（スイート green・強度記録・gaps 記録など、種類に応じて）

### 機能 Issue

- ページ・API・テーブル・副作用出力の一覧
- **「この機能が新規に実装する API」と「依存する横断 API（どの Issue で実装されるか）」を分けて書き**、再実装しない
- Issue 内は**ページ単位でフェーズを分ける**（1 ページを作り切って比較してから次へ）
- ワークフロー: `golden-dataset`（フェーズ B）→ `parity-suite` → `parity-replace` ⇄ `parity-diff`

### 横断 API Issue

- 対象リソースと API 一覧、利用側の機能（fan-out）
- 画面が無くても旧 API の実応答が正解になるため、消費側の機能を待たずに仕様は確定する。**パリティスイートは一度だけ書いて共有する**（機能ごとに重複して書かない）

### バッチ Issue

- 対象バッチ、入力（ゴールデンデータセット＋入力ファイル）、比較する出力（DB 状態・生成ファイル）
- ワークフロー: 姉妹スキルのバッチモード（`parity-suite` が現行出力をベースライン捕捉 → `parity-replace` が新側実装 → `parity-diff` が決定論的比較）

## issues モードのフロー

1. `.replace/features.md` を読み、**未起票**（Issue 番号が空）の機能・横断 API リソース・バッチを列挙する。`--feature <slug>...` があればそれで絞り、無ければ対話選択させる。
   features.md が無い場合は `setup` 未完了として起票せず停止し、`setup` の実行を促す
2. ゴールデンデータセット Issue が未起票なら、選択に関わらず**最初の起票対象として提案する**（すべての前提のため）
3. 起票対象の一覧（種類・slug・タイトル案・依存関係）を提示し、**まとめて承認を得る**
4. 承認後、**1 件ずつ `issue-create` スキルへ委譲する**（本文ドラフトは上記構成で本スキルが用意する）。依存順に起票し、後続の本文から先行 Issue 番号を参照できるようにする
5. 起票された Issue 番号を `.replace/features.md` へ書き戻す
6. 重複チェック（既存 Issue の確認）は**ページネーションに留意する**。issue-create 側の重複チェックが既定件数で打ち切る場合に備え、本モードでも起票前に全件を確認する:

   ```bash
   gh api "repos/$OWNER/$REPO/issues?state=all&per_page=100" --paginate \
     --jq '.[] | select(.pull_request | not) | [.number, .title] | @tsv'
   ```

7. 完了時、起票した Issue の実施に必要な姉妹スキル（`golden-dataset` / `parity-suite` / `parity-replace` / `parity-diff`）が未インストールなら `gh skill install` の手順を案内する
